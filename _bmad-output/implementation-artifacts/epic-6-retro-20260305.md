# Epic 6 Retrospective: Offline Support & Sync

**Date:** 2026-03-05
**Epic Status:** Completed
**Next Epic:** Epic 7 (Settings, Export, and Polish)

## 📊 Epic 6 Review

### What Went Well
- **Offline-First Resilience:** The core objective of allowing cataloging without a network connection was successfully met. The choice to leverage the existing `useItemStore` with MMKV persistence for saving `LocalDraft` objects meant we didn't have to reinvent the state management architecture.
- **Robust Background Syncing:** The `syncService.ts` was implemented with significant forethought, including an `isSyncing` concurrency guard, sequential processing to manage network/backend load, and exponential backoff retry mechanisms for failed syncs.
- **User Experience (UX) Transparency:** Excellent UI feedback mechanisms were built (`OfflineBanner`, `SyncStatusBar`). These components not only communicate network status clearly but also provide interactive elements (animated entry/exit, tap-to-retry on sync errors).
- **Hooks Architecture:** Encapsulating network monitoring (`useNetworkStatus`) and sync triggering (`useSync`) into custom hooks kept the component tree clean and reactive to environment changes.

### Challenges & Struggles
- **Test Coverage Deficit:** A recurring blocker was the absence of a testing framework (e.g., Jest). The background sync logic (`syncService.ts` and `useSync.ts`) is highly complex and asynchronous, and the lack of unit/integration tests led to extended efforts fixing race conditions and edge cases discovered during QA and Code Review.
- **Code Review Iterations:** Multiple code review cycles were necessary to address medium-to-high severity issues, such as race conditions during sync, double-tap risks on UI elements, potential image storage leaks (from uncleaned temp files), and ensuring drafts were properly user-scoped before syncing.
- **Data Model Friction in UI:** Navigating to an `ItemDetailScreen` from a pending `LocalDraft` on the dashboard surfaced limitations. The detail screen expects synced items with Firestore IDs and complete data, causing feature degradation when interacting with drafts.

### Lessons Learned
- **Complex Logic Requires Automated Testing:** Relying solely on manual testing or code review for complex, asynchronous state machines (like a background sync engine) is inefficient and risky. A testing framework is a prerequisite for robust offline-first capabilities.
- **Ephemeral State Handling:** Explicitly separating persisted state (drafts, items) from ephemeral state (sync progress indicators) within the Zustand store was a key architectural win that prevented UI bugs across app restarts.
- **UI Degradation Must Be Planned:** When building offline capabilities, the UI must gracefully handle incomplete data states (like `LocalDrafts` missing Firestore IDs) across all screens, not just the list view.

## 🔄 Previous Action Items (from Epic 5) Update
