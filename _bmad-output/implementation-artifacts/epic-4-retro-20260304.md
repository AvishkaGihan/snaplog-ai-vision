# Epic 4: Item Review, Edit & Save - Retrospective

## Epic Summary
Epic 4 focused on the core data creation and modification flows: reviewing AI-generated data, manually editing fields, saving new items to Cloud Firestore, and deleting existing items. This epic successfully bridged the gap between the AI analysis pipeline established in Epic 3 and the persistent data layer, enabling the complete "Scan to Save" user journey. Key features implemented include the `ReviewFormScreen` with keyboard-aware inline editing, image upload handling, the `firestoreService` for CRUD operations, and the `EditItemScreen` and `ItemDetailScreen` placeholder implementations for future epics.

## What Went Well (Successes)
- **Seamless Data Handoff:** The navigation contract established in Epic 3 (`imageUri`, `aiResult`, `storagePath`) proved robust, allowing `ReviewFormScreen` to seamlessly handle both AI-populated and manual-entry flows with identical code paths.
- **Architectural Adherence:** The architecture rules strictly separating UI state (`useState`), global state (`useItemStore`), and external services (`firestoreService`) were meticulously followed, resulting in clean, testable screen components.
- **Refined UX Details:** Features like keyboard-aware scrolling, return-key field advancement, haptic feedback on save, and standardized snackbar notifications significantly elevated the form's usability.
- **Graceful Resource Cleanup:** The discard flow successfully implemented fire-and-forget cleanup for temporary local images and orphaned Cloud Storage uploads, preventing resource leaks without blocking user navigation.

## What Didn't Go Well (Challenges/Obstacles)
- **Race Conditions in Async Flows:** Several race conditions were identified during the implementation of the save and upload flows:
  - The Back button on `ReviewFormScreen` remained active while `isSaving` was true.
  - The Close button on `CameraScreen` remained active during compression and analysis.
  - The success Snackbar flashed and disappeared because navigation reset instantly.
- **Edge Case Resource Leaks:** Hardware back-button presses and swipe-to-go-back gestures bypassed the standard discard cleanup logic in both Camera and Review screens. The "Retake" flow also initially leaked temporary image files.
- **Incomplete Cleanup Error Handling:** A scenario where Firestore save failed but the image was already uploaded to Cloud Storage resulted in orphaned images.

## Lessons Learned
- **Hardware Navigation is First-Class:** We cannot rely solely on custom button `onPress` handlers for cleanup logic. React Navigation's `beforeRemove` event listeners are mandatory to catch hardware back presses and swipe gestures correctly.
- **Navigation Resets Hide Snackbars:** When using `navigation.reset()` or `navigation.popToTop()`, any UI feedback shown on the current screen (like a Snackbar) will be instantly unmounted. A small timeout (e.g., 1500ms) is required to allow the user to read the feedback before navigating.
- **Atomic Operations Require Manual Rollbacks:** In client-server architectures without distributed transactions, multi-step operations (e.g., Upload Image -> Save Firestore) require explicit `catch` blocks that attempt to rollback the previous steps (e.g., delete the uploaded image if Firestore save fails).

## Action Items & Follow-ups
- **Implement Centralized Error Logger:** While we added silent `console.warn` for cleanup failures, a more robust centralized logging utility (e.g., Crashlytics or a custom logger) should be considered for tracking these background failures in production.
- **Review Previous Action Items:** The action item from Epic 3 regarding removing the `uuid` dependency was not addressed in this epic. We need to ensure action items are carried forward into sprint planning. We will create a chore story for this in the backlog.
- **Enhance Offline Support (Tech Debt):** The current `firestoreService` implementations rely entirely on optimistic local updates via Zustand. We need to verify Firestore's offline persistence behavior and consider how connection state affects the "Save" button availability.
